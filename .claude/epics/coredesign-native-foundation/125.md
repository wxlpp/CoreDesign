# Task: 视觉终审与修复

## Description

跑起预览宿主，逐屏截图，交 `ios-visual-reviewer` 判定观感是否原生，然后修。

这是本 epic 唯一能验证「贴近 Apple 设计风格」这个目标是否达成的环节——前七个任务的验收标准全是可 grep、可编译、可断言的机械判据，但「看着像不像原生」代码评审看不见，必须看渲染图。

## Acceptance Criteria

- [ ] `scripts/run-preview.sh` 能构建并在模拟器启动预览宿主（依赖 Task #117 / 002 已修好 `App/`）
- [ ] 覆盖全部保留组件的截图产出，浅色与深色各一套
- [ ] 至少覆盖一屏组合场景（多个组件同屏），而非只有孤立组件——观感问题常出在元素并置时的节奏与对齐上
- [ ] 派 `ios-visual-reviewer` agent 基于截图评审，重点核对：
      - 字号层级是否呈现 SF 文本样式的节奏，而非等距 web 字阶
      - 圆角是否连续（squircle）且档位一致，嵌套元素是否与容器同心
      - 触控目标是否达到 44pt 视觉体量
      - 层级是否靠 material + separator 表达，而非阴影堆叠
      - 配色在浅色 / 深色下是否与系统 UI 协调
- [ ] 评审结论无「像网页控件 / 非原生」类阻断项
- [ ] 配色在**增强对比度**（Increase Contrast）开启时仍可辨——iOS 系统色免费获得，但 Task #120 的 macOS `windowBackgroundColor` 降级与 accent 的调制衍生族未必满足
- [ ] 阻断项与技术正确的修改建议全部处置完毕，处置后重新截图复审
- [ ] **复审上限 2 轮**：2 轮后仍有阻断项则停下，把剩余项升级给用户裁决，不再自行回卷（与仓库 `auto-fix-pr-after-implementation` 的 3 轮上限同理——无界的「处置→复审」循环允许无限回改 Task #119/004 的定案）
- [ ] 修复后 `swift build` / `swift test` 与 CI 全部命令仍全绿


### 从 #120 交接的观感翻转点（评审指出，测试抓不到，须目视确认）

- [ ] `ListRow.swift:96-100` hover 背景 `surfaceCanvasSubtle`：浅色模式从「比画布**深**」（`#F3F0EA` on `#FCFBF7`）翻转为「比画布**浅**」（`#FFFFFF` on `#F2F2F7`）——hover 变提亮，与旧值及浅色模式惯例相反
- [x] ~~`Badge.swift` neutral badge 背景浅色模式变纯白~~ —— **#122 已定案并修复**：iOS 模拟器实测确认浅色下 badge 与 `surfaceBase` 同为 `#FFFFFF`，改用 `secondaryFill`（填充色专为叠加设计，半透明），并加 `SurfaceContrastTests` 守卫。视觉终审时仍可顺带确认观感，但不再是待决问题。
- [ ] `SegmentedControl.swift:185` thumb 变白（对齐 `UISegmentedControl`，可能反而更好，但未经目视确认）
- [ ] accent 四档（base / hover / pressed / disabled）在浅色与深色下的两两可辨性；`accentDisabled = accent.opacity(0.35)` 在深色纯黑画布上是否还看得见
- [ ] `surfaceCanvasInset` → `tertiaryFill` 后由不透明变半透明，叠在不同底色上结果不同
- [ ] **macOS 目视**：`surfaceCanvas`（`windowBackgroundColor`）与 `surfaceRaised`
      （`controlBackgroundColor`）在浅色与深色下是否**可见地**不同——#120 的单元测试只能断言
      两者是不同的 catalog color，无 WindowServer 会话时解析不出真实 RGBA，「可见差异」这一条
      AC 至今只有符号级证据。这是本清单里唯一的 macOS 项，其余均为 iOS 模拟器视角。

## Technical Details

评审要看的是**渲染结果**，不是代码。截图必须来自真实运行的模拟器，不能用 Xcode Preview 的静态渲染代替——后者在 material、glassEffect、系统色解析上与真机运行有出入。

若评审暴露的是 token 档位选错（而非组件用错档位），修复应回到 token 层改数值，再让 Task #122 的迁移映射表同步更新，而不是在组件里打补丁绕过。

**回改 token 值会波及下游文档**：Task #126 的 `BREAKING-CHANGES.md`「同名换值」表与 `DESIGN-FOUNDATION.md` 取值理由都以终态数值写就。本任务排在 010 之前正是为此——但若本任务改了值，仍须确认 006 的映射表同步更新，否则 010 拿到的输入是旧的。

## Dependencies

依赖 Task #124（全部代码改动与注释清理完成）。实际可运行性依赖 Task #117 / 002 对 `App/` 的修复。

**本任务排在文档定稿（Task #126）之前**：视觉评审看的是模拟器截图，不需要 `docs/snapshots/` 就绪；反过来若文档先定稿，本任务一旦回改 token 值就会让刚写好的「同名换值」表与取值理由当场失真。
