# Task: 同名换值逐点重审与迁移映射表

## Description

Task #121 之后代码是绿的、warning 是干净的——但**最大的一类风险恰恰在这时完全不可见**。

`CoreRadius.small` 从 3 变 6、`medium` 从 6 变 10、`CoreControlMetrics.height(.regular)` 从 32 变 44、语义色改指系统色：这些都是同名换值，编译器不报错、不产生 warning、调用点静默继承新值。尤其新 `small = 6` 恰好等于旧 `medium`——所有按旧语义选 `small = 3` 的调用点圆角直接翻倍。

这可能正是想要的效果，但必须是**逐点确认后的结论**，而不是换值的副产品。本任务的核心产出是那份确认记录。

## Acceptance Criteria

### 迁移映射表（本任务的核心产出）

- [ ] 产出映射表，逐调用点记录：文件:行 → 旧档位/旧值 → 新档位/新值 → 确认理由（保持 / 改档 / 已核对视觉可接受）
- [ ] 圆角：覆盖全部 `CoreRadius` 调用点，重点核 `ProgressBar.swift:45,47`、`SearchField.swift:86`、Badge、Tag 等原本选 `small = 3` 的位置
- [ ] 控件高度：覆盖全部 `CoreControlMetrics.height` 消费点，核 `regular` 32 → 44 的连锁布局影响
- [ ] 语义色：覆盖 Task #120 提供的指向变更清单
- [ ] 表交付给 Task #126 作为 `BREAKING-CHANGES.md`「同名换值」一节的输入

### 圆角出口收敛

- [ ] `Sources` 中裸 `RoundedRectangle(` 调用为 0——全部经 Task #119 的 shape helper 或 `ConcentricRectangle`
- [ ] 嵌套于已知容器的元素改用 `ConcentricRectangle`，容器侧按需 `.containerShape(_:)`

### 验证

- [ ] `swift build` / `swift test` 通过，CI 全部命令通过

## Technical Details

判断依据不是「能不能编译」而是「这个元素在 iOS 上该是什么观感」。若某调用点的新档位明显不合适（例如某处需要的就是极小圆角），正确做法是改用别的档位，而不是回头去调 token 数值——token 档位已在 Task #119 定案，个别调用点不合适说明该点选错档，不是标度错。

例外：若发现某档位对**大量**调用点都不合适，那才是标度问题，应回到 Task #119 的产出重新定案并记录。
