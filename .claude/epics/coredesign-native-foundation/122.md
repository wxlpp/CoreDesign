# Task: 同名换值逐点重审与迁移映射表

## Description

Task #121 之后代码是绿的、warning 是干净的——但**最大的一类风险恰恰在这时完全不可见**。

`CoreRadius.small` 从 3 变 6、`medium` 从 6 变 10、`CoreControlMetrics.height(.regular)` 从 32 变 44、语义色改指系统色：这些都是同名换值，编译器不报错、不产生 warning、调用点静默继承新值。尤其新 `small = 6` 恰好等于旧 `medium`——所有按旧语义选 `small = 3` 的调用点圆角直接翻倍。

这可能正是想要的效果，但必须是**逐点确认后的结论**，而不是换值的副产品。本任务的核心产出是那份确认记录。

## Acceptance Criteria

### 复核面实测（编排在派单前测的，注意时效）

| | 调用点数 | 换值 |
|---|---|---|
| `CoreRadius.small` | **8** | 3 → 6，**恰好等于旧 `medium`** —— 这 8 处圆角直接翻倍，是全库风险最集中的一组 |
| `CoreRadius.medium` | **14** | 6 → 10 |
| `CoreRadius.large` | **4** | 12 → 16 |
| `CoreRadius.none` | 2 | 0，未变 |
| `CoreRadius.xLarge` | **0** | 新增档位，**当前零消费** |
| `CoreControlMetrics.height` | **13** | 五档全换（24/28/32/40/48 → 28/32/44/50/56） |

两点提示：

- `xLarge`(22pt) 零消费本身值得判断——它是为「卡片 / 大容器」设的，若全库确实没有该场景，说明标度多了一档；若有场景本该用它却停在 `large`(16)，那是漏迁。
- 上表是派单前实测，**计数会随任务推进漂移**（本 epic 已发生四次）。以逐点看完为准，不以数字对上为准。

### 从 #121 交接的两处待判断

- [ ] **`CoreMenuButton.swift:60-74` 的图标比例是一处已知欠冲**。选型是换值前做的：当时 16/40 ≈ 0.4 恰好命中 SF Symbol 的 40% 经验值。外框变 50pt 后当前值是 16/50 = 0.32（低于经验值），而 `.large` iconSize (20pt) 现在恰好等于 40%——**原结论方向翻转了**。#121 只更正了数字、保持 `.regular` 不变，换哪一档由本任务判断。
- [ ] **`CircularGlassButtonStyle.swift:22-27` 的「38pt 缩到 32pt（实测）」已失效**。那是换值前的经验数据，height 标度已整体更换。结论方向（`.regular` 仍小于 `.large`）不受影响，但**「显著」这个词需要重新掂量**：换值前是 32 vs 40（差 8pt / 25%），换值后是 44 vs 50（差 6pt / 约 13.6%），相对差距明显收窄。数学上 44 < 50 仍成立，但差距是否还构成「显著」是视觉判断，连同具体数值一并重测。

## Technical Details

判断依据不是「能不能编译」而是「这个元素在 iOS 上该是什么观感」。若某调用点的新档位明显不合适（例如某处需要的就是极小圆角），正确做法是改用别的档位，而不是回头去调 token 数值——token 档位已在 Task #119 定案，个别调用点不合适说明该点选错档，不是标度错。

例外：若发现某档位对**大量**调用点都不合适，那才是标度问题，应回到 Task #119 的产出重新定案并记录。
