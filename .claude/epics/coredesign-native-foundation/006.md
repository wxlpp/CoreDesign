# Task: 清理注释

## Description

源码注释里混入了三类不该进源码的内容：内部流程编号（`（审计项 B7a）`、`Issue #93`、`epic ADR #9`、`per epic ADR #16，PR description 必填字段`）、Primer 考据（token 数值出处、`lineSpacing` 推导公式、Primer 标度对照）、以及中英混排（同一段文档注释前半英文后半中文，英文部分多是 Primer 时期遗留）。

第一类对读者——包括库的下游使用者，因为 `///` 会进 Xcode Quick Help——完全无从理解：它们是 git 历史与项目管理制品，不是代码的一部分。

本任务放在 Task 005 之后：先迁移再清理，避免两次改动同一批文件互相冲突；且 Primer 考据要等 token 重铸完成后才谈得上「作废」。

## Acceptance Criteria

- [ ] `grep -rE "审计项|Issue #[0-9]+|epic ADR|per epic|PR description" Sources` 返回 0 行（基线 23 个源文件）
- [ ] `///` 文档注释统一中文；技术术语保留英文原词（Dynamic Type、Liquid Glass、`ConcentricRectangle`、`strokeBorder` 等不硬译）
- [ ] 文件头统一为 `//  <Name>.swift` + `//  CoreDesign`，去掉 `Source of truth: docs/PRIMER_VERSION.md`
- [ ] Primer 考据段落替换为 Apple HIG 依据或直接删除
- [ ] **保留**解释「为何这么写」的设计说明，例如：
      - `BorderModifier` 中 `strokeBorder` vs `stroke` 的取舍（描边向内画、完整落在 shape 范围内）
      - `BottomInputBar.autoFocus` 为何必须放在 bar 自身的 `onAppear` 中执行
      - `CoreBorderlessButtonStyle` 为何加 `Core` 前缀（避免静默解析到 SwiftUI 同名成员）
- [ ] `swift build` / `swift test` 通过

## Technical Details

判断标准是「读者能否只凭本仓库理解这条注释」：

| 删 | 留 |
|---|---|
| `// 渐变 token 层的首个生产消费点（审计项 B7a）` | `/// 用 strokeBorder 而非 stroke：前者向内画线，描边完整落在 shape 范围内` |
| `/// 收敛自原先四份逐字重复的实现（审计项 B5）` | `/// 该枚举是 color / activeColor / disabledColor 的唯一来源，新增 role 应扩展此处` |
| `// 条件逻辑原样保留（审计项 B8h——其前提经实测不成立）` | `// 两个 onChange 的触发时机不同，不能合并` |

右列保留的是**代码的理由**，左列删除的是**变更的出处**。后者属于 git 历史。

注意 `Toast.swift` 的架构说明整段挂在 `epic ADR #9 / #14 / #16` 上——需要重写为不依赖外部编号的自述式说明，而不是简单删掉编号后留下悬空的「per epic」。
