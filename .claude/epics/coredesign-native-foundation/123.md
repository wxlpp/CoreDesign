# Task: 触控目标与 Dynamic Type 验证

## Description

把 token 重铸带来的两条可访问性承诺兑现成测试：44pt 触控下限，以及文字随 Dynamic Type 缩放。

这两条是 PRD 里少数**可以被断言证明**的原生化指标——前面几个任务的判据多是「grep 为 0」这类结构性检查，能证明代码长什么样，不能证明它在设备上的行为对。

## 已记录的偏离：SegmentedControl 每段命中高度 ≈40pt

容器整体是 44pt（容器级断言通过），但每段的真实命中区域被 `CoreSpacing.xxs`(2pt) 四边内缩
（`SegmentedControl.swift:146` 的 `.padding(CoreSpacing.xxs)`，原生路径见 `:374-375` 的
leading/trailing 约束），因此**单段实际可点高度 ≈ 44 − 2×2 = 40pt**，低于 HIG 的 44pt 下限。

黑盒的外框测量看不到这一点——容器仍报 44pt。

**编排裁决：记录为已知偏离，不在本任务修改。** 三点理由：

1. **原生 `UISegmentedControl` 的标准高度是 32pt**——Apple 自己的分段控件就不满足 44pt。
   本库的 40pt 已经优于平台原生，把它强行抬到 44 会让整个控件比系统的高出一大截，
   与「与系统融为一体」这个 epic 目标相冲突。
2. 修法必然要动玻璃胶囊的内缩比例，那是**视觉判断**，不该在一个测试任务里单方面决定。
3. 若真要满足，正确做法是把容器抬到 48pt 而非压缩内缩——那是更大的设计变更，
   应当作为独立议题讨论，不是顺手改。

**交 #125 视觉终审确认**：40pt 的分段在实机上是否好点。若视觉评审认为偏小，
再作为独立变更处理。

> 同类但未处置的：`SearchField` 内层的 focus 热区与清除按钮各自的 `contentShape`
> 作用域小于外框 44pt。组件级 AC 已满足（整行 44pt），且子区域尺寸无法用黑盒渲染
> 独立验证，故判为超出本任务范围。


## 评审补充的偏离记录

### `SearchField` 的聚焦命中区纵向被收窄（评审 C2）

`contentShape` + `onTapGesture` 挂在**内层** HStack（放大镜 + TextField），而 12pt 纵向 padding 与
`frame(minHeight: 44)` 加在**外层**。`contentShape` 固定的是施加那一刻的几何，祖先的 padding / frame
不会回溯放大它——视觉上 44pt 高的胶囊，上下各约 12pt 的边缘条点下去没有反应。

内层收窄是**有意的**（注释写明：不含尾部 clear button，避免清空后容器 tap 立即重新聚焦），但那个
意图只需要水平方向排除，纵向被一起收窄是副作用。

**不在本任务修**：改动手势架构属交互设计变更，且无法用黑盒渲染验证修复效果。交 #125 实机确认。

> 这条同时暴露了 `TouchTargetTests` 原本的方法论声明有误——它称 bounding box 高度即命中区域高度，
> 那只对 `contentShape` 挂在最外层的组件成立。声明已更正，例外清单已补全。

### `BottomInputBar` 中段未被验证（评审 I1）

它是三个独立可点控件的 HStack。整行高度 ≥ 44pt 被两侧 44–50pt 的圆形按钮平凡满足，
中间 textFieldContainer 自身的激活区域没有被验证。同 `SearchField`，交 #125。

### `Toast` 点击关闭区无法测试（评审 I2）

`ToastView` 是 `private`，`@testable import` 只暴露 `internal`，测试无法直达。其
`contentShape` 施加在 `.padding` 与 `.floatingGlass` 之后（作用域正确），静态阅读判断应满足 44pt，
但无断言支撑。若要覆盖，需先把它提为 `internal`——那是 API 表面变更，不在本任务范围。
