# Task: 文档、CI 与版本收尾

## Description

把地基更换的结果落到文档、CI 与版本上。这里包含一件容易被当成「以后再说」的事：`CLAUDE.md` 与 `AGENTS.md` 里有整节关于主题系统与 CoreGradient 的说明，Phase 1 之后全部变成错误指引——而这个仓库重度依赖 agent 按这两份文件工作，过期指令有实际成本。

## Acceptance Criteria

### 设计依据文档

- [ ] `docs/PRIMER_VERSION.md` 删除，替换为 `docs/DESIGN-FOUNDATION.md`
- [ ] `DESIGN-FOUNDATION.md` 记录 Apple HIG 依据与各 token 取值理由，含 Task #120 提供的 accent 衍生族档位取值理由

### 破坏性变更清单

- [ ] `docs/BREAKING-CHANGES.md` 新增本次条目，含三节：
      - **删除的公开符号**：6 个组件、`StatusResult`、`CoreGradient`、`Blossom` trait、`CoreRadius.smallPlus/mediumPlus`、`CoreControlMetrics.primerVerticalPadding`、`CoreTypography` 的 `*LineSpacing`/`*Tracking`/`fixedFont` 等
      - **改名的 token**：`CoreTypography` 的 9 个改名档位（旧名 → 新名）、10 个旧 `*Font` static var、以及 `caption` / `captionMono` 的**同名换语义**与 `subheadline` 净新增
      - **同名换值**（独立成节，探针对此失明）：`CoreRadius` 各档、`CoreControlMetrics.height` 各档、语义色指向变更，逐项列旧值 → 新值
- [ ] **删除符号一节必须逐项覆盖以下已确认清单**（由 #117 / #118 实际删除，不得遗漏）：
      - Issue #117：`BookCover`、`RefPill`、`StatusRow`、`EventRow`、`CommentCard`、`TimelineItem` 六个组件；`StatusResult` 枚举；`timelineDepth` 的 `EnvironmentValues` 入口
      - Issue #118：`Blossom` package trait（下游若写 `.package(url:..., traits: ["Blossom"])`，升级后在**依赖解析期**报 unknown-trait，报错发生在 manifest 层而非编译层，下游未必能关联到本次变更，须明写）；`CoreGradient.brand` / `.cta` / `.canvas` 三个公开符号，替代映射为 `brand`/`cta` → `Color.accent`、`canvas` → `Color.surfaceCanvas`
      > 之所以钉在这里而不是让 #117 / #118 各自往 BREAKING-CHANGES 写：两个 PR 同时改该文件会在 epic 分支撞车；而「公开 API 已删但无记录」的窗口结构上不存在——epic→main 只在含本任务在内的全部任务完成后才发生。
- [ ] 替代物要等 `0.4.0` 才有的条目明确标注「将于 0.4.0 提供」，不留空
- [ ] 说明下游可评估从 `0.2.0` 直接跳到 `0.4.0`

## Technical Details

BREAKING-CHANGES 的「同名换值」一节是本任务最容易做浅的部分。下游对 `CoreRadius.small` 从 3 变 6 是编译无感的，若不逐项列出，升级后只会表现为「界面看着不太对」而无从定位。该节内容来自 Task #122 的迁移映射表。

## Dependencies

依赖 Task #125——文档必须以**视觉终审后的终态数值**写就。Task #122 的迁移映射表是 BREAKING-CHANGES「同名换值」一节的输入；若 009 的修复回改了 token 值，本任务须以修复后的值为准。
