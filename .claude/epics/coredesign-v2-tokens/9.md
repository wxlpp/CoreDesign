# Task: FocusRingModifier macOS NSFocusRing 分支填充

## Description

在 Task 7 创建的 `FocusRingModifier.swift` macOS 分支（`#if canImport(AppKit)`）中，用 `NSViewRepresentable` 包装一个空 NSView，通过 `focusRingType = .exterior` + `becomeFirstResponder` 同步 SwiftUI focus state，注册到系统 NSFocusRing 机制。

**这是 platform API spike**，PRD 已写好回退条款：若 NSFocusRing 集成失败 **或** Swift 6 strict concurrency 不兼容，回退为 macOS 也走 Task 7 的 `.overlay(stroke())` 视觉近似实现，并在任务 progress comment 中记录原因。

## Acceptance Criteria

### 主路径（系统集成成功）

- [ ] `#if canImport(AppKit)` 分支用 `NSViewRepresentable` 包装空 `NSView` 子类
- [ ] NSView 子类配置：`focusRingType = .exterior`、`canBecomeKeyView = true`
- [ ] `makeNSView` / `updateNSView` 满足 Swift 6 strict concurrency 要求：
  - `@MainActor` 隔离正确
  - `becomeFirstResponder` / `resignFirstResponder` 调用无 non-Sendable 跨 actor 警告
  - 任何外部状态传递通过 `@MainActor` 同步路径
- [ ] 通过 SwiftUI `@FocusState` 或外部传入的 `visible: Bool` 驱动 `becomeFirstResponder` / `resignFirstResponder`
- [ ] 替换 Task 7 留下的 macOS 占位 overlay 实现
- [ ] 提供 macOS `#Preview`
- [ ] **人工验证**：在 macOS 上运行带 `.focusRing()` 的 view，Accessibility Inspector 扫描时该焦点环识别为 system focus indicator，**不是** generic visual element

### 回退路径（spike 失败时）

- [ ] 在任务 progress comment 中明确记录失败原因：是 NSFocusRing 行为不符预期，还是 Swift 6 并发不兼容
- [ ] macOS 分支沿用 Task 7 的 `.overlay(RoundedRectangle().stroke())` 视觉实现（即不动 Task 7 的 macOS 占位）
- [ ] PRD SC #11（macOS Accessibility Inspector 识别）改为"已回退，限制已记录"
- [ ] 不阻塞 epic 关闭

## Technical Details

- 参考 AppKit 文档：`NSView.focusRingType`、`NSView.becomeFirstResponder()`、`NSResponder.canBecomeKeyView`、`NSView.draw(_ dirtyRect:)`
- Swift 6 strict concurrency 关注点：
  - `NSViewRepresentable.makeNSView` / `updateNSView` 是 `@MainActor` 隔离的
  - `becomeFirstResponder` 跨 SwiftUI state propagation 周期调用时若涉及 non-`Sendable` 值会触发 strict concurrency 错误
  - 必要时在 `Coordinator` 类上加 `@MainActor`
- **时间盒**：0.5 天内决定走主路径还是回退；不要陷入超过 0.5 天的 NSFocusRing 调试。spike 失败立即触发回退路径，剩余时间归还
