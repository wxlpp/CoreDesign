# Task: 视觉终审——实机复刻一屏 iOS 设置页

## Description

本 epic 全部交付物是**视觉组件**——机械判据(编译/测试/grep)证明不了它们好不好看。这是唯一能验证「与系统融为一体」的环节,是硬门不是可选(ADR-4)。

Phase 1 的视觉终审(#125)一次抓出四个只有实机截图才发现的缺陷:深色 Badge 语义色 α 太低不可见、预览宿主 320×480 兼容模式、`.preferredColorScheme` 泄漏、Tag 命中区把 chip 撑大 2.1 倍。本任务同等对待。

## Acceptance Criteria

- [ ] 在预览宿主里搭一屏**用 `InsetGroupedSection` + `SettingsRow` 复刻的 iOS 设置页**,不写任何 CoreDesign 之外的样式代码(PRD Success Criteria #10 / 本 epic Success Criteria #5)
- [ ] 预览宿主确认全屏渲染(`AXFrame` = 目标机型全尺寸,非 320×480 兼容模式)——Phase 1 已修好 `INFOPLIST_KEY_UILaunchScreen_Generation`,但每次都要确认
- [ ] 深色 + 浅色各一套截图,覆盖:设置页整屏、`Card`、5 个 `.core` style 控件、`.core` toggle 的 `.tint` 覆盖效果
- [ ] **最大辅助功能字号(`.accessibility5`)抽查**:设置页在最大字号下不裁切、不重叠(PRD NFR;002 的 SectionHeader/Footer 与 004 的 5 个 style 没有各自的字号测试,在这里兜底目视)
- [ ] 派 `ios-visual-reviewer`,重点核对:
  - 分组容器是否浮在画布之上(不是与画布同色——这是 001 要修的塌缩的最终验收)
  - `SettingsRow` 的图标方块、分隔线 leading inset 是否对齐 iOS 设置惯例
  - `.core` style 五个控件深浅双模式下的观感,尤其 `.tint` 覆盖是否真的生效
  - 触控目标(实机点,不只看图)
- [ ] 评审结论无「非原生 / 不可见 / 分隔线错位」类阻断项;技术正确的修改建议全部处置,处置后重新截图复审
- [ ] **复审上限 2 轮**,超出停下升级给用户裁决(同 Phase 1 #125)
- [ ] 修复后 `swift build` / `swift test` / iOS 腿仍全绿

## Technical Details

截图必须来自真实运行的模拟器,不能用 Xcode Preview 的静态渲染代替——后者在 material、glassEffect、系统色解析上与真机有出入(Phase 1 证实)。

Phase 1 已有可用工具:`scripts/run-preview.sh`、`axe`(坐标是点非像素)、`xcrun simctl ui <UDID> appearance dark|light`。注意 demo app 的 `ComponentDetail` 两栏对照现在用 `.environment(\.colorScheme,)`(Phase 1 修好了 `.preferredColorScheme` 泄漏),可信。
