# Task: 修 SurfaceKind 表面色塌缩

## Description

这是 Phase 2 的**地基任务**,不是新组件——但后面所有容器类组件(`Card`、`InsetGroupedSection`)都建在它之上,不修则它们在深色下隐形。

Phase 1 视觉终审(#125)与 issue #136 查明:`.surface(.content)` → `surfaceCard` → `surfaceCanvas` → `systemGroupedBackground`,即 `.content` 背景与页面画布**完全同色**。iOS 卡片本该浮在画布之上(`secondarySystemGroupedBackground`),而库里 `surfaceRaised` 已指对,只是 `.content` / `.card` 两个 case 指错了。

## Acceptance Criteria

- [ ] **改 `surfaceCard` token 的别名目标**:`SurfaceColors.swift` 里 `surfaceCard` 从别名 `surfaceCanvas` 改为别名 `surfaceRaised`(= `secondarySystemGroupedBackground`)。这样 `.content` / `.card` 两个 `SurfaceKind` case 自动跟随(它们都取 `surfaceCard`),不必碰 `private` 的 `SurfaceKind.background`
  > **为何取这条而非改 `SurfaceKind` case**(评审定死,不再二选一):`surfaceCard` 全仓只被那两个 case 消费、`.surface(.content/.card)` 生产调用点为零,两案净效应相同——但改 token 别名 (a) 可测:`Color.surfaceCard` 是公开 token,守卫能直接断言;(b) 一并修掉「公开 token `surfaceCard` = 画布色、注释却写卡片」这个名实相悖的公开陷阱;(c) 对下游是公开 token 同名换值,BREAKING-CHANGES 表述干净
- [ ] **改写 `SurfaceColors.swift:97-98` 的刻意注释**:现注释写「卡片刻意保持接近画布,只靠边框和相邻 panel 拉开层级」——这是 Phase 1 的设计,#125/#136 视觉终审推翻了它(深色下与画布同色、隐形)。改为记录新决策:卡片用 raised 背景浮在画布之上,符合 iOS 分组容器惯例
- [ ] **真守卫测试**:断言 `Color.surfaceCard != Color.surfaceCanvas` 在**深色**下成立(`Color.resolve`,`#if os(iOS)`,跑 iOS 腿——macOS 无 WindowServer 时会塌缩成同 fallback)。复用 Phase 1 `SurfaceContrastTests` 模式。**注意**:token 级断言 `surfaceRaised != surfaceCanvas` 今天不改代码就为真(见 `SystemBackgroundColorsMacOSTests.swift:36-41`),那不是守卫;必须断言 `surfaceCard` 本身,才锁得住这次修复
- [ ] 审视 9 个 `SurfaceKind` 是否收敛档位(#136 第 1 条:实际只 3 个背景)——**收敛是可选项**,若做则在交付说明记录新映射;不做也要说明为何留待后续
- [ ] `docs/BREAKING-CHANGES.md` 的 `0.4.0` 条目记录 `.content`/`.card` 指向变更(**同名换值**,对下游编译零感知)

## Technical Details

修法已定死(见 AC 第 1 条):改 `surfaceCard` token 别名,不碰 `SurfaceKind.background`。

实测消费面(评审已核):`surfaceCard` 全仓只被 `SurfaceModifier.swift:46,53` 两个 case 消费;`.surface(.content/.card)` 生产调用点为**零**(唯一 `.surface(` 调用是 `ListRow.swift:98` 的 `.canvas`,**不受影响**,列表行本就该贴画布)。所以这个改动当前无任何生产组件受影响,纯粹是为 Phase 2 的 `Card` 铺好可见的地基。
