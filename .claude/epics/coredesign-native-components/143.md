# Task: 系统控件 .core style 套件

## Description

FR-12。给 `Toggle` / `TextField` / `ProgressView` / `Label` / `DisclosureGroup` 各配一个 CoreDesign style,经对应 style 协议的静态成员 `.core` 暴露。**不重新实现控件本身**——这是「不重造系统控件」约束的正解:用系统控件 + 换皮,而非重写。

## Acceptance Criteria

- [ ] `.toggleStyle(.core)` / `.textFieldStyle(.core)` / `.progressViewStyle(.core)` / `.labelStyle(.core)` / `.disclosureGroupStyle(.core)` 各一
- [ ] 每个 style 保留系统控件的**状态绑定 + 可访问性语义**;视觉可换。**注意「保留全部原生行为」对 Toggle 与 TextField 不可字面达成**:自定义 `ToggleStyle.makeBody` 会整体替换原生 switch 的手势与触感(拖动滑块、haptic),`TextFieldStyle._body` 是私有的——这两个若无法在不重造控件的前提下既改视觉又保原生行为,如实报告(见下方「宁可少一个诚实的 style」)。`ProgressView` / `Label` / `DisclosureGroup` 的 `makeBody` 全公开、`.tint` 接入无障碍、库内有先例,无此风险
- [ ] **强调色必须走 `.tint` 通路(ADR-3 硬约束)**:`makeBody` 中不得写死 `Color.accent`,须经 `TintShapeStyle`(`.tint`)或 configuration 取。测试或实机证明:对控件施加 `.tint(.red)` 后强调色确实变红。参考 `ProgressIndicator` / `ProgressBar` 已有的 tint 接入
- [ ] 库不重新实现这些控件本身(不出现自造的 `CoreToggle` 之类)
- [ ] 各 style 同文件 `#Preview`,深浅双模式,且演示 `.tint` 覆盖效果
- [ ] `public` 表面完整
- [ ] **圆角一律经 `CoreShape` / `ConcentricRectangle`,不出现裸 `RoundedRectangle(`**(PRD SC#3,Phase 1 已把 Sources 清零,新组件别重新引入)
- [ ] `swift build` / `swift test` 通过;iOS 腿覆盖(若测试在 `#if os(iOS)` 下)

## Technical Details

按 SwiftUI 惯例:每个 style 实现对应协议(`ToggleStyle` / `TextFieldStyle` / …),然后在协议的 `where Self == CoreXxxStyle` 扩展上加 `static var core`。形态参考 Phase 1 的 `ButtonStyle` 扩展(`static func *Button`)与 `BannerStyle`。

`.tint` 接入是本任务的核心风险点,也是 FR-12 存在的理由。`TextFieldStyle` 的 `_body` 是私有的,SwiftUI 对它的自定义能力有限——若某个 style(尤其 TextField)无法在不重造控件的前提下既改视觉又保留 tint,如实报告,不要为了凑齐 5 个而重造控件。宁可少一个诚实的 style,不要一个假的。
