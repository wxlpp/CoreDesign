# Task: 系统控件 .core style 套件

## Description

FR-12。给 `Toggle` / `TextField` / `ProgressView` / `Label` / `DisclosureGroup` 各配一个 CoreDesign style,经对应 style 协议的静态成员 `.core` 暴露。**不重新实现控件本身**——这是「不重造系统控件」约束的正解:用系统控件 + 换皮,而非重写。

> **实际交付见 AC:3 个 style（`progressView` / `label` / `disclosureGroup`）+ 2 个诚实略过（`toggle` / `textField`，理由见 AC）。** 原始范围列 5 个,但 Toggle/TextField 无法在不重造控件的前提下换皮,按任务逃生条款如实略过。

## Acceptance Criteria

- [x] **实际交付 3 个 style**:`.progressViewStyle(.core)` / `.labelStyle(.core)` / `.disclosureGroupStyle(.core)`。**`.toggleStyle(.core)` 与 `.textFieldStyle(.core)` 诚实略过**(见下,非漏做)
  - **Toggle 略过理由**:自定义 `ToggleStyle.makeBody` 整体替换原生 UISwitch,拖动手势与 haptic 必丢;补回等于重造控件(AC 明令禁止)。
  - **TextField 略过理由**:`TextFieldStyle` 唯一 requirement 是下划线私有的 `_body`,无公开第三方自定义入口;能做的只有 `.coreTextField()` View modifier 组合(可视可换但满足不了 `.textFieldStyle(.core)` 的字面 AC)——作为后续任务的选项报给 epic。
- [x] 每个交付的 style 保留系统控件的**状态绑定 + 可访问性语义**;视觉可换。(DisclosureGroup 换皮后系统不再自动播报展开态,已显式补 `.accessibilityValue`;ProgressView 百分比走本地化 `format: .percent`。)
- [x] **强调色必须走 `.tint` 通路(ADR-3 硬约束)**:三个 style 的 `makeBody` 均无 `Color.accent` 字面量,填充/icon/chevron 全经 `.tint`。iOS 腿 `CoreControlStyleTintTests` 以 ImageRenderer 像素级验证 `.tint(.red)` 通道占优。
- [x] 库不重新实现这些控件本身(不出现自造的 `CoreToggle` 之类)
- [x] 各 style 同文件 `#Preview`,深浅双模式,且演示 `.tint` 覆盖效果
- [x] `public` 表面完整
- [x] **圆角一律经 `CoreShape` / `ConcentricRectangle`,不出现裸 `RoundedRectangle(`**(PRD SC#3,Phase 1 已把 Sources 清零,新组件别重新引入)
- [x] `swift build` / `swift test` 通过;iOS 腿覆盖(`CoreControlStyleTintTests` 3 tests 绿)

## Technical Details

按 SwiftUI 惯例:每个 style 实现对应协议(`ToggleStyle` / `TextFieldStyle` / …),然后在协议的 `where Self == CoreXxxStyle` 扩展上加 `static var core`。形态参考 Phase 1 的 `ButtonStyle` 扩展(`static func *Button`)与 `BannerStyle`。

`.tint` 接入是本任务的核心风险点,也是 FR-12 存在的理由。`TextFieldStyle` 的 `_body` 是私有的,SwiftUI 对它的自定义能力有限——若某个 style(尤其 TextField)无法在不重造控件的前提下既改视觉又保留 tint,如实报告,不要为了凑齐 5 个而重造控件。宁可少一个诚实的 style,不要一个假的。
