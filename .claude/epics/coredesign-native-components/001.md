# Task: 修 SurfaceKind 表面色塌缩

## Description

这是 Phase 2 的**地基任务**,不是新组件——但后面所有容器类组件(`Card`、`InsetGroupedSection`)都建在它之上,不修则它们在深色下隐形。

Phase 1 视觉终审(#125)与 issue #136 查明:`.surface(.content)` → `surfaceCard` → `surfaceCanvas` → `systemGroupedBackground`,即 `.content` 背景与页面画布**完全同色**。iOS 卡片本该浮在画布之上(`secondarySystemGroupedBackground`),而库里 `surfaceRaised` 已指对,只是 `.content` / `.card` 两个 case 指错了。

## Acceptance Criteria

- [ ] `SurfaceModifier.swift` 的 `SurfaceKind.background`:`.content` 与 `.card` 从 `surfaceCard`(= 画布色)改指 `surfaceRaised`(= `secondarySystemGroupedBackground`)
- [ ] 或:直接把 `surfaceCard` token 从别名 `surfaceCanvas` 改为别名 `surfaceRaised`——二选一,取影响面更小的一种,在交付说明里写明选了哪条及理由
- [ ] 守卫测试:`.content`(经 `surfaceRaised`)与 `.canvas`(经 `surfaceCanvas`)在**深色**下 `Color.resolve` 不同色。复用 Phase 1 的 `SurfaceContrastTests` 模式,`#if os(iOS)`,跑在 iOS 腿(macOS 无 WindowServer 时会塌缩成同 fallback)
- [ ] 审视 9 个 `SurfaceKind` 是否收敛档位(#136 第 1 条:实际只 3 个背景)——**收敛是可选项**,若做则在交付说明记录新映射;不做也要说明为何留待后续
- [ ] `docs/BREAKING-CHANGES.md` 的 `0.4.0` 条目记录 `.content`/`.card` 指向变更(**同名换值**,对下游编译零感知)

## Technical Details

修改点二选一:
1. 改 `SurfaceKind.background` 的 `.content` / `.card` case(影响这两个 kind 的所有消费点)
2. 改 `surfaceCard` token 的别名目标(影响所有引用 `surfaceCard` 的地方)

先 grep 两者的实际消费面再决定。`ListRow` 用的是 `.surface(.canvas)`(**不受影响**,列表行本就该贴画布),所以 `.content` 的现有消费点需单独核查——很可能当前零生产消费(Phase 1 视觉终审提过 `.surface(.content)` 无生产调用点),那样这个改动纯粹是为 Phase 2 的 `Card` 铺路。
