# Task: 色彩层重组

## Description

承载 12 个审计项：A1、A2d、B1a、B1b、B1c、B6a、B6b、B6c、D11、D13、D14、D19（对应 FR-1）。

核心动作是**把第 4 层色彩的职责收窄为「状态功能别名」**（epic 的 AD-1）：删除 `FunctionalColor` 的 `primary`/`secondary`/`tertiary` 三组，交互色统一走第 3 层 `InteractionColors`。这一个动作同时解决四件事——A1 遮蔽 bug、库内外语义分裂、与 `InteractionColors` 逐值重复的色阶、以及一处零效果的 Blossom 分流。

**本任务最大的技术风险是 A1 型静默重解析**：`Color.primary/secondary/tertiary` 遮蔽 SwiftUI 同名成员，删除这些符号时**编译器不报错**，只会静默把解析目标改到 SwiftUI 内建成员。因此必须用下述**毒丸机制**让编译器穷举残留使用点，而不是靠「逐处显式改写」这种过程纪律。

## Technical Details

### A1 毒丸机制（必做，不可替代）

三处被遮蔽符号的引用（`CheckBox.swift:31`、`StatusRow.swift:80`、`CoreGradient+Preview.swift:17`）在符号删除后**不报错**，只会静默改解析到 SwiftUI 内建成员。

**做法**：先不删符号，而是加一个中间 commit，给 `FunctionalColor` 的 12 个遮蔽符号全部标注：

```swift
@available(*, deprecated, message: "A1 probe: shadows SwiftUI builtin; use contentPrimary or an explicit token")
```

编译一次，编译器会**精确报出全部残留使用点**（含审计未发现的）。逐处改写至零诊断后，再删符号。

**必须用 `deprecated`，严禁用 `unavailable`。** epic 已实测四种方案：

| 方案 | 结果 |
|---|---|
| `@available(*, unavailable)` | **零诊断**，静默通过。重载决议把 unavailable 候选**排除**出候选集，直接落到 SwiftUI 内建成员 |
| 改名（`primary` → `__a1_primary`） | 零诊断，同样静默落到 SwiftUI |
| 移出 `Color` 扩展到独立命名空间 | 零诊断，同上 |
| **`@available(*, deprecated)`** | **精确报出每个使用点**：`warning: 'primary' is deprecated: A1 probe` |

差别在于 `deprecated` 候选**仍参与重载决议并胜出**，只是附带诊断；`unavailable` 候选被踢出决议。越"强"的标记对本场景越无效。

**硬闸门**：配合 `-warnings-as-errors` 可让残留点无法被忽略（实测输出 `error:` 而非 `warning:`）：

```bash
swiftc -typecheck -warnings-as-errors ...   # 或临时给 target 加 .unsafeFlags(["-warnings-as-errors"])
```

### `statusAccent*` 保留（原判「整组删除」已改判）

**改判理由**：删除它与本任务自身的 B6a（`Banner`/`Toast`/`Badge` 迁出 legacy）**直接冲突**。新体系只有 `accent` 一个蓝色家族，而它正是 Primer 的 info 语义——那三个组件的 legacy `info*` 只能迁到它。D19 原判据「库内零渲染消费点」恰恰是因为 info 当时走 legacy，迁移一做该判据即失效。

**D19 同时改判**：原记「`statusAccentEmphasis` 的 light 值是 accent 单组的 colorset 笔误」。横向比对五组后发现 `accent`/`success`/`attention`/`danger`/`done` 的 `*-emphasis` light 值**逐组等于同组 `*-muted`**（`#DDF4FF`/`#AFF5B5`/`#F8E3A1`/`#FFC1BA`/`#DACDFB`），是整个新体系的系统性错误。Primer 语义里 emphasis 是饱和实色。

**改为**：修正五组 emphasis 的 light 值为各组 `fg` 同色（accent `#0969DA`、success `#1F883D`、attention `#9A6700`、danger `#CF222E`、done `#8250DF`），四个 `status-accent-*.colorset` **保留不删**。

### legacy status 组迁移（四处，漏一处即编译失败）

`Toast` / `Badge` / `Banner` / **`Form`** 全量迁移到新体系。**`Form.swift:101` 使用 legacy `Color.dangerForeground`，`:92,99` 的文档注释同样引用——漏改会直接编译失败**，这是最容易被忽略的一处（`Form` 不在直觉上的「status 组件」列表里）。

`Badge` 仍留在 legacy 的直接原因是新体系缺 `*Border` 档（B6c），故补档是迁移的前置。

### `borderFocus` → `.accent` 的视觉代价

通过**别名继承**实现 Blossom 跟随，与 `borderSelected` 的既有处理一致，使全库分流点净减（9 → 8）。

代价：**默认主题下 focus ring 从 Primer 蓝 `#0969DA` 变为品牌蓝 `#0077FA`**（dark 从 `#1F6FEB` 变 `#3295FB`），影响两个真实渲染点：`FocusRingModifier.swift:106`（默认参数）与 `SearchField.swift:136`。这是已在 NFR 中声明的视觉例外第 3 条，不是回归。
