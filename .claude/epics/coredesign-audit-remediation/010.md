## Technical Details

### spec 结构体收敛模式（B8f）

`StateLabel` 的 4 个 switch 与 `StatusRow` 的 3 个 switch 都在对同一个枚举做多次穷举，每次取一个维度（颜色 / 图标 / 文案 / 背景）。收敛形态是引入一个私有 spec 结构体，单个 switch 返回它：

```swift
private struct Spec { let icon: String; let tint: Color; /* ... */ }
private var spec: Spec { switch self.style { /* 单次穷举 */ } }
```

收益不只是行数——新增 case 时编译器只在**一处**要求穷举，漏配某个维度的可能性归零。

### `StatusLevel` 合并的波及面（B8e）

`ToastLevel` 与 `MessageLevel` 合并后，`Toast` 与 `Banner` 两个组件的公开签名都会变。这是 breaking change，按 epic 决策**直接改，不留 deprecated 兼容层**。须同步更新：仓库内 `App/` 预览宿主、`docs/components/` 下引用这两个枚举的示例、以及测试文件中的构造点。

### `Badge` / `Tag` 是 init 形态的好范例（D6a、D6b）

`Badge.swift:83` 与 `Tag.swift:80` 已经是「String 便利 init + `@ViewBuilder` designated init」的双层形态。D6a / D6b 的目标就是让 `StateLabel`、`StatusRow`、`EventRow`、`SidebarNavigationRow` 对齐到同一形态——**便利 init 保留**（现有调用点不必全改），designated init 新增。

### `Sidebar.swift` 的额外验证要求（关键）

D6b 会重塑 `SidebarNavigationRow` 的 init 形态，而 **004 在 `Sidebar.swift` 上落了 Dynamic Type 布局断言**。该断言被 `#if os(iOS)` 包住，**四条 SwiftPM 命令下完全不可见**——在 macOS 宿主里编译都不进入。

因此本任务**必须额外跑第 5 条验证命令**：

```bash
xcodebuild test -scheme CoreDesign -destination 'platform=iOS Simulator,name=iPhone 17 Pro'
```

不跑这条，弄破布局断言将无人发现（epic「每个 Issue 的完成定义」第 4 条明确点名本任务）。若 001 的 CI 降级到本地 pre-push 级别，这条命令须在本地手工执行并记录输出。
