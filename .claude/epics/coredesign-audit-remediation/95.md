# Task: Dynamic Type 改造

## Description

承载 2 个审计项：B2a、D3（对应 FR-3）。**项数最少，但改动面是全 epic 最大的单项**——触及 28 文件（排除 #6 整删的 `EmptyState.swift` 后实际改 27）。

问题：`Tokens/CoreTypography.swift:53-183` 全部 10 个字体 token 用 `.system(size:weight:)`，`relativeTo:` 出现 0 次。全库仅两处组件级适配。更反常的是——**token 层越是被正确使用，Dynamic Type 越失效**：7 处绕过 token 直接用系统 TextStyle 的文字恰好因此还能缩放。这对一个对外分发的设计系统是可访问性硬伤。

改造方向（epic AD-3）：`CoreTypography` 从 `Font` 常量升级为 `CoreTypography.Token` 枚举 + `.coreFont(_:)` modifier，内部用 `ScaledMetric(wrappedValue:relativeTo:)` 在 init 中按 token 角色指定缩放基准，保住 Primer 精确基准字号的同时恢复缩放，并把文档本就建议的「font / lineSpacing / tracking 三件套」收进单一调用点。

**本任务同时编写布局断言层**（不在 #7）——安全网必须与它守护的改动同批合入。

## Acceptance Criteria

- [ ] `CoreTypography` 升级为 `CoreTypography.Token` 枚举 + `.coreFont(_:)` modifier，**9 个 token 支持 Dynamic Type 缩放 + `captionSmall` 明确固定**（B2a；原写「10 个全缩放」，`captionSmall` 的 9pt 紧凑 chrome 场景有既有设计约束「故意不缩放」，见 `CoreTypography.swift:172`——放大会撑爆 tab 角标/status bar。`coreFont` 仍支持它，只是走固定路径。#95 执行期经用户确认）

## Technical Details

### 触及集（28 文件，实际改 27）

- **24 个 `CoreTypography.` 消费者**（94 处，与 PRD FR-3 一致）
- **`RefPill`**——D3 的 `.caption.monospaced()`
- **`AvatarGroup`**——D3 的 `:59` `.caption2`（该文件零 `CoreTypography` 引用，故不在 24 之内）
- **`SolidButtonStyle` / `LightButtonStyle`**——`font(for:)` → `fontToken(for:)`

排除 #6 整删的 `EmptyState.swift` 后实际改动 **27 文件**。

`Sidebar.swift` 有 **16 处** `CoreTypography.` 引用，是全库最高的单文件消费者——#5 的 B5 骨架收敛完成后会缩到约 6 处，这是把本任务排在 #5 之后的收益之一。

### `CoreControlMetrics.font(for:)` → `fontToken(for:)`（关键，不可省）

`CoreControlMetrics.swift:174-179` 的 `font(for:)` 返回 `Font` 且被全部按钮样式消费。而 `@ScaledMetric` 需要 View 上下文，modifier 形态服务不了返回 `Font` 的 API。

改为 `CoreControlMetrics.fontToken(for:) -> CoreTypography.Token`，调用方写：

```swift
.coreFont(CoreControlMetrics.fontToken(for: controlSize))
```

**不这样做则按钮文字仍不缩放，SC-3 名不副实。**

### `RefPill` 的 mono 变体（为保持现有表现所必需）

`RefPill.swift:37,40,45` 用 `.caption.monospaced()`，而 `CoreTypography` 中 `monospaced` 出现 0 次。须**新增 mono 变体 token**，否则迁移会静默丢失等宽特性（ref / branch 名的视觉回归）。

此项属「为保持现有表现所必需」，不受 Out of Scope「不做加法」约束。

### 硬顺序：`.coreFont()` 先落地，7 处系统字号迁移后进行

D3 的 7 处（`AvatarGroup.swift:59`、`StatusRow.swift:46`、`StateLabel.swift:50`、`CommentCard.swift:56`、`RefPill.swift:34,37,40,42,45`、`BottomInputBar.swift:302,376`）当前是 TextStyle，**今天会随 Dynamic Type 缩放**。顺序颠倒会让它们经历「先失去缩放再恢复」的中间态。

（「处」按调用簇计：`RefPill` 的 5 行同类调用计为 1 处，`BottomInputBar` 的 2 行分属两个上下文计为 2 处；按行数则为 11 行 / 6 文件。）

NFR 视觉例外第 5 条：`.subheadline` 为 15pt，无精确对应 token，迁移必有小幅字号变化。

### 布局断言层（由本任务编写，不在 #7）

epic 明确：安全网必须与它守护的改动同批落地。若放在 #7，本任务完成时第 5 条命令下无任何 `#if os(iOS)` 测试可跑，等于空转——而 Dynamic Type + Sidebar 裁切正是全 epic 风险最高的改动。

**编写约束（全部已实测确认）**：

- **`#if os(iOS)` 包住**——macOS 无 Dynamic Type：`@ScaledMetric(wrappedValue: 16, relativeTo: .body)` 在 macOS `swift test` 宿主下全 12 档**恒返回 16.0**（`NSFont.preferredFont(.body).pointSize` 恒为 13.0）；iOS 26 Simulator 上正常（`large=16` → `xxxLarge=21` → `accessibility5=45`）
- **经 `xcodebuild test -destination 'platform=iOS Simulator,name=iPhone 17 Pro'` 运行**（NFR 第 5 条命令）。前四条 SwiftPM 命令下自动跳过
- **相邻档位断言用 `>=`**——实测 iOS 上 `small` 与 `medium` 同为 15pt，严格 `>` 会假失败
- **跨档（`large` vs `xxxLarge`）才用 `>`**
- **不得依赖颜色**——`swift test` 下 asset 颜色解析为 `(0,0,0,0)`（SwiftPM 不调用 `actool`，产出 bundle 只有原始 `.xcassets` 目录）
- 断言机制：用 `ImageRenderer` 在注入的 `dynamicTypeSize` 下渲染并断言高度单调增长且 ≥ 内容固有高度

**下游提醒**：布局断言被 `#if os(iOS)` 包住，四条 SwiftPM 命令下不可见。凡后续触及被覆盖文件的 Issue（至少 `Sidebar.swift`——#10 的 D6b 重塑 row init、#11 的 D18 补 Preview）同样须跑第 5 条命令，否则弄破断言无人发现。

### 范围边界

`CoreControlMetrics.iconSize(for:)` 驱动的图标字号（如 `CheckBox.swift:37,41`）**不纳入** Dynamic Type 改造，保持固定尺寸。US-2 承诺的是文字缩放，执行时不要自行扩大到图标。

`Sidebar` 四处 `frame(height:)` → `minHeight`（B2b）**归 #5 实施**，本任务不做——它与 #5 的 Sidebar row 骨架收敛改同一批行。
