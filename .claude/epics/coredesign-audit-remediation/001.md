# Task: 构建配置前置：CI、defaultIsolation、预览宿主 trait

## Description

本任务是整个 epic 的**硬前置**，承载 5 个审计项：C1、C7a、C7b、C9a、C9b。

它做三件事：

1. **`Package.swift` 启用 `.defaultIsolation(MainActor.self)`（C7a）**——这个开关**改变全库编译语义**，所有文件在新的隔离默认值下重新检查。若它晚于 #2–#11 落地，期间写的代码会在开关合入时批量报错。**这是本 epic 唯一的硬性顺序约束：本任务须最先合入。**
2. **CI 基建（C1）**——`.github/workflows/` 目前是空目录。须验证 GitHub Actions runner 的 Xcode 26 / iOS 26 Simulator 可用性，按四级降级决策树落地门禁。
3. **构建配置修正（C7b、C9a、C9b）**——`Package.swift:9` 的 `.iOS("26.0")` 改枚举形式；`App/project.yml` 的 `xcodeVersion: "16.0"` 与 iOS 26 部署目标自相矛盾，须修正；预览宿主未声明 `traits`，Blossom 在唯一视觉验证载体里看不到，须补 `traits: ["Blossom"]`。

CI 部分**不阻塞任何其它 Issue**——其余 Issue 的「验证绿」以 NFR 的本地命令为准，CI 降级只影响门禁形态，不影响验证标准。但 `defaultIsolation` 部分是所有后续 Issue 的前提。

## Technical Details

### `defaultIsolation` 的 fallout 约束（关键风险，须严格遵守）

全库默认隔离翻转**可能在任意组件或测试文件冒出并发诊断**，owner 矩阵没有建模这一点。因此修复须受以下约束：

- **最小化**：只修让构建变红的诊断，不做任何顺带整理
- **只加注解不重构**：用 `nonisolated` / `@MainActor` / `Sendable` 注解解决，不改结构、不搬代码
- **不碰 #6 删除名单内的文件**：`EmptyState.swift`、`Utils/View+SizeReader.swift`、`Utils/KeyboardHandling.swift`（及 `KeyboardHandlingTests.swift`、`EmptyStateDeprecationTests.swift`）即将被 #6 整文件删除。若它们冒诊断，用最小注解让它编译通过即可，**不做任何整理**，避免白改
- **超预期即停**：诊断量若明显超出「零星几处注解」的预期，**停下回报**而非硬修——大面积并发诊断意味着需要重新评估该开关的落地方式，不应由本任务独自吞下

### CI 的四级降级决策树（按序尝试，任一成功即停）

1. `macos-26` hosted image（若已 GA）
2. `macos-15` + `xcodes` / `xcode-select` 装载 Xcode 26
3. self-hosted runner
4. 全部不可行 → 降级为**本地 pre-push 脚本**跑五条命令作为临时闸门，CI 留待 runner 就绪后回补

**每一级都要同时验证 runner 提供可用的 iOS 26 Simulator**——NFR 第 5 条命令（`xcodebuild test -destination 'platform=iOS Simulator,name=iPhone 17 Pro'`）依赖它，而 #4 的布局断言层正是靠该命令运行。仅有 Xcode 26 而无 iOS 26 Simulator 的 runner 不满足要求，须继续降级。

若最终降级到第 4 级，须在任务记录中明确：SC-1（下游 probe）与 SC-4（CI workflow）随之降级为本地执行，#4 的布局断言层将失去自动化守护。
